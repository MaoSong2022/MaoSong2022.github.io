## Data Parallelism

data parallesim (DP) 的核心思想是

<Note title="Core Idea" emoji="💡" variant="info">
  将模型复制到多个 GPU 上，每个 GPU 同时运行不同的 micro batches.
</Note>

### Naive DP

假如我们现在要训练一个模型 $f(x,y;\theta)$, 其中 $x,y,\theta$ 分别是输入，输出和训练参数。
给定训练集合 $\mathcal{D}=\{(x_i,y_i)\}_{i=1}^N$, 我们的训练目标如下

$$
\mathcal{L}(x, y;\theta, \mathcal{D}) = \frac{1}{N}\sum_{i=1}^N f(x_i, y_i;\theta)
$$

使用 DP 之后，假如我们现在有 $G$ 个 GPU, 我们将模型 $f(x,y;\theta)$ 复制到这 $G$ 个 GPU 上，得到 $f_1=\dots=f_G=f$, 然后每个 GPU 就可以处理大小为 $N_G:= N/G$ 的 micro batch:

$$
\mathcal{L}(x, y;\theta, \mathcal{D}) = \frac{1}{G}\sum_{i=1}^G\mathcal{L}(x, y;\theta, \mathcal{D}_i)=\frac{1}{G}\sum_{i=1}^G\frac{1}{N_G}\sum_{j=1}^{N_G} f_i(x_j,y_j;\theta)
$$

但是这也意味着不同的 GPU 实例上有不同的 loss 和梯度，第 $i$ 个 GPU 上的梯度为

$$
\nabla_\theta\mathcal{L}(x, y;\theta, \mathcal{D}_i) = \sum_{i=1}^G\frac{1}{N_G}\sum_{j=1}^{N_G} \nabla_\theta f_i(x_j,y_j;\theta)
$$

为了解决模型同步的问题，我们会使用 [AllReduce](#all-reduce) 来统一反向传播

$$
\mathrm{AllReduce}([\nabla_\theta\mathcal{L}(x, y;\theta, \mathcal{D}_1),\dots,\nabla_\theta\mathcal{L}(x, y;\theta, \mathcal{D}_G)])
$$

这样每个 DP rank 上的模型更新就是

$$
f_i(x,y;\theta_{k+1}) = f_i(x,y;\theta_{k}) - \alpha\nabla_\theta\mathcal{L}(x, y;\theta, \mathcal{D}),i=1,\dots,G
$$

从而我们保证了模型每一次更新之后，在不同的 DP rank 上都是一致的。

### Optimization

Naive DP 的逻辑非常简单，不同的 DP rank 完成 forward pass, backward pass, AllReduce, 权重更新。
但是在 AllReduce 的过程中，我们发现 GPU 出现了 bubble, 也就是 idle, 我们需要对其进行优化。

注意到，我们在进行反向传播的时候， 其实并不需要等待反向传播完成再进行 AllReduce, 而是可以在某一个参数完成反播之后立马就进行AllReduce,
也就是说，我们将粗粒度的model level AllReduce 细化到了 parameter level AllReduce.

下面是对应的实现

<Accordion title="Optimization on Naive DP">

```python
class DataParallelNaive(nn.Module):
    def __init__(self, module):
        super().__init__()
        self.module = module
        self.require_backward_grad_sync = True
        self.register_backward_hook(self._all_reduce_grads)
    
    def forward(self, *inputs, **kwargs):
        return self.module(*inputs, **kwargs)

    def register_backward_hook(self, hooks):
        for p in self.module.parameters():
            if p.requires_grad is True:
                p.register_hook(hook)

    def _all_reduce_grads(self, grad):
        if self.require_backward_grad_sync:
            dist.all_reduce(grad, op=dist.ReduceOp.SUM, group=pgm.process_group_manager.cp_dp_group)
            grad /= pgm.process_group_manager.cp_dp_word_size
        
        return grad
    
    @contextlib.contextmanager
    def no_sync(self):
        self.require_backward_grad_sync = False
        yield
        self.require_backward_grad_sync = True
```

### ZeRO

在上一节我们介绍了如何使用多个 GPU 来实现 DP.
我们这样做有一个内在假设：模型权重可以被放到单个 GPU 里。
但是，现在大部分模型都不能放到一个 GPU 里，一个 80GB HBM 内存的 GPU 最多只能放下一个 40B (BF16精度) 的模型，而现在大部分模型都到了 $\sim100$ B的级别。
因此，我们需要对现有的 DP 算法进行优化。

第一个算法就是微软提出的 ZeRO [@ZeRO]. ZeRO 的核心思想为

<Note title="Core Idea" emoji="💡" variant="info">
  将训练时的模型参数，梯度，优化器状态复制到多个 GPU 上，来减少 DP 的内存占用。
</Note>

对于 Naive DP, 我们的每一个 DP rank 里都保留一份完整的参数，梯度，优化器状态用于前向，反向传播。
可以看到，不同的 DP rank 实际上存储的东西是彼此的副本，这显然带来了很大的冗余。
因此， ZeRO 就想办法来避免这些冗余。

假设模型参数量为 $\Psi$, 在训练过程中，我们需要保存以下信息：

1. 模型参数： BF16 精度, $2\Psi$
2. 模型梯度： BF16, $2\Psi$
3. master weights: FP32, $4\Psi$
4. optimizer states: Adam, FP32, $4\Psi+4\Psi$
5. (optional) accmulcated gradients, FP32, $4\Psi$

本节中，我们暂时不考虑 accumulated gradients, 因此整体的内存消耗就是 $2\Psi+2\Psi+12\Psi$.

ZeRO 的思想就是将这些不同的信息切分到不同的 DP rank 上进行分块保存，进而降低内存消耗。

假设我们现在有 $N$ 个 GPU, 我们分别来介绍不同的 ZeRO 算法

#### ZeRO-1

对于 ZeRO-1 来说，我们会把 optimizer states 均匀切分到 $N$ 个 GPU上，这样每个 GPU 保存的就是 $k\Psi/N$ (这里我们用 $k$ 来表示不同 optimizer 对应的系数，对 Adam 来说是
$12$).

训练时，我们通过 AllGather 来收集不同 GPU 上的 optimizer states 再进行计算：

1. 前向传播计算 loss
2. 反向传播计算梯度
3. 进行一次 ReduceScatter 在不同 GPU 之间同步梯度
4. 每个 GPU 进行一次 optimizer step, 得到局部更新的 optimizer states 以及权重更新量
5. 对权重更新量进行一次 AllGather 来更新模型参数

#### ZeRO-3

ZeRO-3 则进一步将模型参数也进行了 sharding, Pytorch 将 ZeRO-3 的原生实现称之为 Fully Sharded Data Parallelism (FSDP) [@FSDP; @SimpleFSDP].

ZeRO-3 在 forward pass 中，先使用 AllGather 收集所需参数再进行计算，计算完之后，收集的参数会被立即释放。
Backward pass 时，也是一样的道理

ZeRO-3 的计算流程为：

1. 前向传播时，对每一层，先通过 AllGather 收集当前层的参数进行前向传播，计算完毕之后然后立刻通过 flush 释放掉当前层的参数
2. 反向传播时，对每一层，先通过 AllGather 收集当前层的参数进行反向传播，然后
   1. 计算完毕之后立刻通过 flush 释放掉当前层的参数
   2. 通过 ReduceScatter 将梯度同步到不同的 GPU 上
3. Optimizer step, 每个 GPU 上单独更新本地的参数

#### Analysis on ZeRO

按照通信来对比，我们可以得出如下结果

|         Strategy         |       naive DP      |             ZeRO-1            |             ZeRO-2            |             ZeRO-3            |
|:------------------------:|:-------------------:|:-----------------------------:|:-----------------------------:|:-----------------------------:|
| Forward Pass             | None                  | None              |  None             |  AllGather Parameters $\Psi$         |
| Backward Pass            | ReduceScatter gradients   $\Psi$     | ReduceScatter gradients   $\Psi$             |  ReduceScatter gradients   $\Psi$             |  AllGather Parameters $\Psi$ +ReduceScatter gradients   $\Psi$         |
| Optimizer Step           | None                  | None              |  None            |  None         |
| Parameter Update         | None                  | AllGather Parameters $\Psi$              |  AllGather Parameters $\Psi$              |  None         |
| Total                    | $\Psi$                 | $2\Psi$              |  $2\Psi$              |  $3\Psi$         |

实际上，如果我们更改 ZeRO-1 和 ZeRO-2 的参数更新时间，将其放在下一次迭代的 forward pass 开始时，我们就会发现，ZeRO-1/2 相比与 ZeRO-3， 实际上的区别仅在于 backward pass 时是否需要 AllGather parameters. ZeRO-1 和 ZeRO-2 保存了一份完整的参数，因此不需要 AllGather, 而 ZeRO-3 为了极致的内存优化，每次利用完参数之后就会直接释放掉，这导致其在反向传播时需要一次额外的 AllGather 来进行计算。
这体现了时间换空间的思想。
